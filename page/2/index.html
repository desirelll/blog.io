修改 zoo.cfg 配置

本例子是在三个服务器上分别跑一个 ZooKeeper
所以需要你有三个服务器（电脑或者云服务器）
首先看一下 zoo.cfg 的配置

  • 因为是不同服务器，所以启动端口可以是一样的 clientPort=2181
  • dataDir 是本服务器的数据文件夹，可以自己指定，然后必须在该文件夹内创建一个名为 myid 的文件，里面的内容就是服务器的 id。比如在第一台服务器（192.168.1.2）中，myid 的内容就为 1，没有其他任何字符。
  • 最后三行就是三个服务器的 IP 地址和端口号，端口 2888 由对等 ZooKeeper 服务器使用，用于互相通信；而端口 3888 用于领导者选举。您可以选择自己喜欢的任何端口。通常建议在所有 ZooKeeper 服务器上使用相同的端口。

综上所述，三台服务器中的 ZooKeeper 文件夹中不同的地方就是 dataDir 的路径以及 dataDir 中的 myid 的内容（在这里分别为1、2、3）

运行 & 测试

修改完成之后逐个启动即可
./zkServer.sh start
然后在其中一个服务器中启动客户端连接其他服务器的地址

Zookeeper 伪集群部署

发表于 2017-09-05 | 分类于 java ， ZooKeeper | | 阅读次数

伪集群部署即在同一个服务器上开启多个 ZooKeeper 服务，现实中肯定不会这样使用，因为要是这个服务器挂了，那这些 ZooKeeper 就都挂了，这样就没有分布式的意义了。但在资源有限的情况下可以使用这种方法来学习、测试多服务器的部署，这里以3个 Server 为例

创建一个文件夹 ZooKeeper，并在里面放置三个 ZooKeeper，改名为 server2，server3，server4

修改 zoo.cfg 文件

  • server2/conf/zoo.cfg
  • server3/conf/zoo.cfg
  • server4/conf/zoo.cfg

可以看到三个配置文件不同的地方是

  • dataDir 目录：分别在自己的文件夹内创建一个 data 文件夹
  • clientPort：因为是在同一个服务器下，所以三个启动端口必须不一样

创建 myid 文件

在上面指定的 dataDir 里面分别创建一个名为 myid 的文件（没有后缀名），存储的是各自的 server id

  • /Library/zookeeper/server2/data/myid
  • /Library/zookeeper/server3/data/myid
  • /Library/zookeeper/server4/data/myid

启动服务

分别进入 bin 文件夹内执行
./zkServer.sh start
并执行
./zkCli.sh -server 127.0.0.1:端口号
注意端口号要用到配置文件中的三个不同的端口号，以便测试所有服务都可用，完了我们就已经开了三个连接不同端口的客户端
然后在其中一个执行 create /test hello 指令创建一个节点，在另外的客户端中执行 ls /命令查看是否创建成功即可

netty 基本使用（四）

发表于 2017-08-28 | 分类于 java ， netty | | 阅读次数

处理基于流的数据传输

一个关于套接字缓冲区的注意事项

在上一节中我们遇到了一个奇怪的数组越界错误，如果对流传输没有一个比较清楚的了解，可能会不知道为什么会出现这样的错误
在像 TCP/IP 这样基于流的数据传输中，接收到的数据是存储在一个套接字缓冲区(socket receive buffer)中的，然而基于流传输的缓冲区不是一个数据包队列，而是一个字节队列。这意味着当你发送两条独立的消息包的时候，他们将不会被当做两条消息来处理，而是当成一堆字节，所以不能保证所见即所得
比如说当操作系统的 TCP/IP 堆栈收到了三个数据包

但有很大的可能性你在你的应用程序上看到的是酱紫的

所以对接收方来说，不管是服务器还是客户端，都应该对数据进行整理以便其看起来是符合逻辑的，在这个例子里接受到的数据应该被整理成酱紫的

第一种解决方案

回到我们的 TimeClient，就能明白出现错误的原因了，我们那可怜的32位的整数在传输时变得支离破碎了
第一种解决方案就是创建一个内部累积缓冲区(internal cumulative buffer)，并等待直到收到四个四节的数据

  1. ChannelHandler 有两个生命周期监听器方法：handlerAdded（）和handlerRemoved（）。 只要不长时间阻止，您可以执行任意初始化/销毁任务。
  2. 收到的数据需要累积到 buf 里
  3. 然后，处理程序必须检查buf是否有足够的数据，在此示例中为4个字节，然后继续执行业务逻辑。 否则，Netty会在更多数据到达时再次调用channelRead（）方法，最终所有4个字节都将被累积。

第二种解决方案

虽然第一种方案能够正确解决问题，但是 handler 类看起来不简洁，如果实现的业务太复杂的话，用这种方法写的代码就会变得很难维护。
可能你也已经知道了我们可以往 ChannelPipeline 添加多个 ChannelHandler，所以可以把一个 ChannelHandler 拆分为多个以减少程序的复杂性。在这里我们可以把 TimeClientHandler 拆为两个

  • 用来处理分片问题的 TimeDecoder
  • 初始简单版本的 TimeClientHandler

刚好 netty 提供了一个可扩展的类能供我们使用

  1. ByteToMessageDecoder 是 ChannelInboundHendler 的是一个实现类，主要用来处理分片问题
  2. 每当接受到数据时，ByteToMessageDecoder 就会调用带有内部维护累积缓冲区(internally maintained cumlative buffer) 的 decode（）方法
  3. 当累积缓冲区中的数据还不完整时，就不会往 out 添加任何东西
  4. 当 decode（）往 out 添加了一个对象，说明 decoder 解码一个消息成功。记住我们不需要多次解码消息，因为 ByteToMessageDecoder 会一直调用 decode（） 方法直到没有数据往 out 中添加了

  1. NioEventLoopGroup 是一个处理 I/O 操作的多线程事件循环，因为这里要实现的是一个服务端应用程序，所以需要有两个 NioEventLoopGroup。第一个“老板” NioEventLoopGroup 用来接收收到的连接。第二个“员工” NioEventLoopGroup 是当“老板”接收到连接时并将接受的连接注册给“员工”之后，用来处理接受的连接的传输。使用多少线程以及它们如何映射到创建的通道取决于 EventLoopGroup 实现，甚至可以通过构造函数进行配置
  2. ServerBootstrap 是用来设置一个服务器的帮助类，你可以直接使用 Channel 来设置服务器，但这是一个繁琐的过程，一般都不这样使用。
