API-blueprint 学习

发表于 2018-01-17 | 分类于 tools ， API | | 阅读次数

最近在进行接口服务开发的过程中，一开始搭建的项目是适用 swagger 来自动化生成 API 文档，但由于在开发过程中没有使用实体类，直接使用 map 来传递参数，导致生成文档时没办法显示接口所需要的具体参数（swagger要求传入实体类来获取参数），因此学习了 API-blueprint
Github 地址: API-blueprint

简介

API-blueprint 是用来编写 API 文档的一种标记语言，基本上就是用 Markdown 来写的，再配合一些工具把文档渲染成 HTML 文件。

另外，使用 API-blueprint 的一个好处是可以使用 mock 测试，即在后端接口还没开发完时，通过数据模拟让前端人员也可以同步开发测试，等接口写完后再联调即可。

API-blueprint

使用 API-blueprint 的第一步是指定 API 的名称和 Metadata：

Metadata

API-blueprint 以一个 metadata 块开始，用来指定 FORMAT 的值为 1A，表示API-blueprint 的版本。

API 名称和描述

第一个标题就是 API 的名称，在上面的例子中就是 “Polls”。标题以一个或多个 # 开头，# 的数量表示标题的等级。

标题下的句子就是 API 的描述信息，可以继续使用标题来分割描述信息。

  • 懒汉式的特点是延迟加载，比如配置文件，直到被用到时才会加载。

Zookeeper 伪集群部署

发表于 2017-10-26 | 分类于 java ， ZooKeeper | | 阅读次数

伪集群部署即在同一个服务器上开启多个 ZooKeeper 服务，现实中肯定不会这样使用，因为要是这个服务器挂了，那这些 ZooKeeper 就都挂了，这样就没有集群的意义了。但在资源有限的情况下可以使用这种方法来学习、测试多服务器的部署，这里以3个 Server 为例

创建一个文件夹 ZooKeeper，并在里面放置三个 ZooKeeper，改名为 server2，server3，server4

修改 zoo.cfg 文件

  • server2/conf/zoo.cfg
  • server3/conf/zoo.cfg
  • server4/conf/zoo.cfg

可以看到三个配置文件不同的地方是

  • dataDir 目录：分别在自己的文件夹内创建一个 data 文件夹
  • clientPort：因为是在同一个服务器下，所以三个启动端口必须不一样

创建 myid 文件

在上面指定的 dataDir 里面分别创建一个名为 myid 的文件（没有后缀名），存储的是各自的 server id

  • /Library/zookeeper/server2/data/myid
  • /Library/zookeeper/server3/data/myid
  • /Library/zookeeper/server4/data/myid

启动服务

分别进入 bin 文件夹内执行
./zkServer.sh start
并执行
./zkCli.sh -server 127.0.0.1:端口号
注意端口号要用到配置文件中的三个不同的端口号，以便测试所有服务都可用，完了我们就已经开了三个连接不同端口的客户端
然后在其中一个执行 create /test hello 指令创建一个节点，在另外的客户端中执行 ls /命令查看是否创建成功即可
